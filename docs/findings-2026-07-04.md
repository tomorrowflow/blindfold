## Architecture vocabulary (Ousterhout / deep-module lens)

Findings use the project's own architecture language:

- **Module** ‚Äî anything with an interface + implementation (function, class, slice).
- **Deep** ‚Äî a small interface hiding a lot of behaviour (high *leverage*).
- **Shallow** ‚Äî interface nearly as complex as the implementation.
- **Seam** ‚Äî where an interface lives; a place behaviour can be swapped without editing in place. **One adapter = hypothetical seam; two adapters = real seam.**
- **Deletion test** ‚Äî imagine deleting a module: if complexity *vanishes* it was a pass-through; if it *reappears across N callers* it was earning its keep.
- **Locality** ‚Äî change/bugs/knowledge concentrated in one place (what maintainers get from depth).

Domain terms follow `CONTEXT.md` (entity, surrogate, mapping, blindfold/restore/re-identify, L1/L2/L3, closed-world restore, verify pass, fail-closed, coherent surrogate world).

## Executive summary

The **cryptographic core is sound** and there are several genuinely **deep modules** worth protecting (`detect_l2`, `StreamingRestorer`, `verify_pass`, `TransitClient`). But three themes cut across the review:

1. **The product's headline differentiator is not wired in.** The coherent surrogate world (ADR-0004/0005 ‚Äî employer-coherent fake email domains, date-shifting, locale pools) lives in a fully-tested `SurrogateEngine` that **no request path constructs**. The live path mints from four disjoint flat pools with none of that coherence. This is the largest doc‚Üîcode drift in the repo (`ARCH-1`).

2. **The whole trust model rests on a spoofable header.** Every RBAC check keys off a self-asserted `x-blindfold-identity` header with no token, signature, or session. Anyone who can reach the socket can grant themselves `admin` and decrypt the crown-jewel mapping (`SEC-1`). Compounding it, several paths write **real entity values** into audit records, 503 bodies, and logs (`SEC-3`, `SEC-4`), and the fail-closed leak check runs **after** the payload has already egressed (`SEC-5`).

3. **A fresh install is a locked, empty room.** Following the README yields a management app that shows nothing and permits nothing: the entity graph is never seeded into the UI's store, RBAC has a chicken-and-egg lockout with no bootstrap admin, and there's no runnable entry point (`uvicorn` isn't even a dependency) (`UX-1`, `UX-2`).

The fastest trust wins are the four cheap leak/audit correctness bugs (`SEC-3`, `SEC-4`, `SEC-5`, `SEC-9`) ‚Äî they contradict the crown-jewel invariants even in single-user mode. The biggest architectural decision is `ARCH-1` (wire the coherent engine or formally defer it).

## 1. Architecture findings

### ARCH-1 ‚Äî The coherent surrogate world is tested but dead ‚Äî **Strong**
**Files:** `surrogate_engine.py` (whole file), `surrogates.py`, `review.py:76`, `store/_mint.py`; ADR-0004, ADR-0005; `CONTEXT.md` ¬ß"Coherent surrogate world".
**Problem:** `SurrogateEngine` (`surrogate_engine.py:93`) implements the differentiator ‚Äî employer-coherent fake email domains, stable per-entity date-shift offsets, locale-aware pools ‚Äî and is referenced **only by its own test** (`test_surrogate_engine.py`). Nothing in `app.py`/`engine.py` imports it. The live request path mints from flat pools (`surrogates.py`, `store/_mint.py`, `review.py`) with **no coherent world, no date-shift, no email coherence**. Yet ADR-0004 titles this "the differentiator" and CONTEXT.md lists it as an invariant. This is the most dangerous kind of green test: full confidence, zero coverage of production behaviour.
**Deletion test:** deleting `SurrogateEngine` breaks only its own test ‚Äî the product loses nothing it currently ships. That *is* the finding.
**Decision:** wire provisional/PII/seed minting **through** `SurrogateEngine` (one deep minting authority consuming the `employer` edge the graph already stores), **or** mark ADR-0004/0005 "not yet implemented" and delete the dead module + `L3Adjudication.kind`. Don't leave it in limbo.

### ARCH-2 ‚Äî Dual L3 seam: one concept split into two half-interfaces ‚Äî **Strong**
**Files:** `app.py:170` (`get_l3_detector`), `app.py:180` (`get_l3_adjudicator`), `app.py:256-294` (`_scan_l3_or_block`), `engine.py:224-243`, `l3.py`.
**Problem:** "When does L3 run?" is split across **two independent dependency seams that both fire on every request**: the learning-loop detector runs *inside* the engine (`engine.py:224`), and a **second, freshly-constructed** `L3Detector` runs again over the whole payload for fail-closed blocking (`app.py:271-275`). Two instances scan the same text for the same concept, with **no shared content cache**, wired from different providers. Tests can only ever exercise one at a time (learning-loop tests override one seam and rely on the other being a no-op; fail-closed tests do the reverse) ‚Äî so "learning loop AND fail-closed in one request" is untestable through the interface.
**Decision:** make the engine's single `L3Detector` the source of truth; have `blindfold_payload` surface the outcome (confirmed / minted / `L3Unavailable`) so the handler makes the fail-closed call from the *same* pass. Collapse the two seams into one. Leverage: the content cache finally amortizes; locality: L3 lives in one module.

### ARCH-3 ‚Äî The seams the domain most wants are hypothetical; the loud one is over-factored ‚Äî **Worth exploring**
**Files:** `app.py:82,154` (`get_mapping`), `store/repository.py`, `store/postgres.py`, `reidentify.py`.
**Problem:** Seam reality check:
- **Store repository** (`seeded_pairs()`): two implementations exist (`VendoredSeedRepository` sync, `PostgresSeedRepository` async) but they don't share a real interface ‚Äî one is `def`, one `async def`, so no caller treats them polymorphically. `get_mapping` hardwires the sync vendored path; **Postgres is not wired into the request path**. Real seam for ETL/tests, *hypothetical* for the proxy.
- **Re-identify store** (`ReIdentificationStore` Protocol): one adapter (`InMemoryReIdentificationStore`), only ever seeded by tests. The Postgres adapter its docstring promises doesn't exist. So `reidentify_surrogate` **always 404s in production**.
- **Upstream provider**: a real network boundary (tests swap httpx via `MockTransport`), but Anthropic vs OpenAI are two *methods*, not two adapters ‚Äî not a polymorphic provider seam.
- Meanwhile the **dual L3** (ARCH-2) is loudly abstracted with three adapters but is over-factored.
**Decision:** define one (async) repository interface both repos satisfy, and give `get_mapping`/`get_reidentify_store` a real production wiring path (ETL populates the re-identify store). Until then the Protocols are documentation, not seams.

### ARCH-4 ‚Äî Four near-identical recursive string/JSON walkers with a load-bearing difference ‚Äî **Worth exploring**
**Files:** `engine.py:160` (`_blindfold_json_value`), `engine.py:312` (`_restore_json_value`), `engine.py:433` (`_collect_into`), `app.py:346` (`_collect_strings`).
**Problem:** The same dict/list/str recursion appears four times. Two of them join leaves differently and it is **privacy-load-bearing**: `engine._collect_text` joins with `\x00` (`engine.py:423`) so a value can't match across fields (verify-pass precision), while `app._collect_text_for_l3` joins with `\n` (`app.py:333`) for L3 sentence heuristics. A naive "dedupe" would silently break either verify precision or L3 quality. The invariant currently lives in a **comment, not a type**.
**Decision:** extract one `walk_string_leaves(value, fn)` primitive; keep `_collect_text` (NUL) and `_collect_text_for_l3` (newline) as thin *named* callers so the join distinction becomes a documented parameter, not a copy-paste hazard.

### ARCH-5 ‚Äî Merge side-effects duplicated across two endpoints ‚Äî **Worth exploring**
**Files:** `app.py:817-852` (`merge_entities`, by canonical name) and `app.py:1089-1117` (`merge_entities_by_id`).
**Problem:** Each endpoint repeats the `mapping.seed(...)` / `retire_surrogate` / `AuditRecord(event="entity-merged")` block. The graph layer already exposes both `merge` and `merge_by_ids` (`entity_graph.py:242/272`); only the HTTP layer duplicates the post-merge effects. (Note: the by-name path also has a leak bug ‚Äî see `SEC-4`.)
**Decision:** one `_apply_merge_side_effects(...)` helper; endpoints differ only in winner/loser resolution (name vs id) and response shaping.

### ARCH-6 ‚Äî Dead minting code and unread fields ‚Äî **Worth exploring**
**Files:** `surrogates.py:80,145,24` (`SurrogateMapping.mint`, `_next_surrogate`, `_SURROGATE_POOL`), `l3.py:83` (`L3Adjudication.kind`).
**Problem:** The non-PII novel-mint path in `surrogates.py` is never called ‚Äî the engine only calls `mint_pii`, and novel entities are minted by the inbox (`review.py:76`). `L3Adjudication.kind` is read nowhere (only `PiiSpan.kind` is used). Deletion test: removing `mint()` breaks only `test_surrogate_mapping.py`.
**Decision:** fold into `ARCH-1`'s resolution ‚Äî if the coherent engine becomes the single minting authority, these die naturally; otherwise delete them explicitly.

**Deep modules to protect (do not "refactor" away):** `detect_l2`/`detect_pii` (`detection.py:141/63` ‚Äî 4-pass matcher behind a 2-arg pure function), `engine.blindfold_payload`/`restore_response`/`verify_pass`, `engine.StreamingRestorer` (`engine.py:339` ‚Äî split-surrogate correctness behind `feed`/`flush`), `l3.L3Detector`, `transit.TransitClient`. These are the high-leverage cores; the test surface over them is healthy (egress asserted on recorded bytes via `MockTransport`, not mock call counts).

## 2. Security findings

> Severity tags follow the security pass. Several items are **acceptable for the single-user localhost dev slice** but are **hard blockers before any multi-user or network-exposed deployment** ‚Äî tagged accordingly. Items `SEC-3/4/5/9` are true leak/audit bugs worth fixing now regardless of deployment mode.

### SEC-1 ‚Äî Management API is unauthenticated; identity is a spoofable header ‚Äî **Critical (blocker before multi-user)**
**Files:** `app.py:642-643` (`_caller_identity`), `app.py:646-651` (`_require_role`), `app.py:704-724` (`grant_workspace_role`), `app.py:855-899` (`reidentify_surrogate`).
**Problem:** `_caller_identity` returns `request.headers.get("x-blindfold-identity", "")` ‚Äî no token, session, signature, or verification anywhere (no `add_middleware`/`Authorization`/`api_key` in the tree). Every RBAC check trusts a self-asserted string. Any caller can assert an identity holding `re-identifier` and **decrypt the crown-jewel mapping**, or assert `admin` and **grant themselves any role** (full privilege escalation). This collapses invariant #3 (re-identify is RBAC-gated) the moment the socket is reachable by anyone but the operator.
**Decision:** what authenticates the caller before RBAC means anything? (mTLS? signed token? reverse-proxy-injected identity the app trusts only from loopback?) This is the gate for every other RBAC finding.

### SEC-2 ‚Äî Dev root token bypasses all Transit policies ‚Äî **Critical (dev-only)**
**Files:** `infra/docker-compose.dev.yml:3,8`, `infra/bootstrap-openbao.sh:29`, `config.py:35`.
**Problem:** Dev runs `-dev-root-token-id=dev-root-token` and the bootstrap defaults `TOKEN=dev-root-token`. A root token bypasses the carefully-separated policies (`bootstrap-openbao.sh:55-80` correctly splits proxy/operator/human capabilities). `config.py:35` reads `BLINDFOLD_OPENBAO_TOKEN` with no guard that it isn't root.
**Decision:** fine for dev/CI; must run under the `blindfold-proxy` policy token in any shared deploy. Consider a startup assertion that rejects a root token when not in an explicit dev mode.

### SEC-3 ‚Äî Real entity value leaks into audit record, 503 body, and logs ‚Äî **High (fix now)**
**Files:** `engine.py:409-413` (verified: `LeakError(f"...: {real!r}")` + `logger.warning`), `app.py:311-319` (`_verify_or_block` puts `{exc}` into `reason`), `app.py:237-252` (`_blocked_response` writes `reason` to audit **and** returns it in the 503 JSON `error.message`).
**Problem:** On a verify-pass leak, the real value flows into (a) the audit log, (b) the client-visible 503 body, and (c) the process log at WARNING. Directly violates invariant #2 ("real-value side never stored in plaintext") and leaks to the client. The `UnresolvedSurrogateError` sibling path only exposes surrogates (safe).
**Decision:** scrub the real value from the exception/reason/log ‚Äî reference the offending entity by surrogate or a hashed id, never the plaintext.

### SEC-4 ‚Äî Merge audit record stores real canonical names ‚Äî **High (fix now)**
**Files:** `app.py:828-838` (`merge_entities`), contrast `app.py:1098-1105` (`merge_entities_by_id`, which correctly logs only IDs).
**Problem:** The by-name merge endpoint logs `reason=f"winner={...canonical_name...!r}, loser={loser_canonical!r}"` ‚Äî real entity names into the audit log, even though the response body just below is carefully scrubbed (`app.py:840-852`). The entity-id endpoint got this right; the name endpoint didn't.
**Decision:** log IDs/surrogates in the merge audit reason, not canonical names. Naturally folds into `ARCH-5`'s shared helper.

### SEC-5 ‚Äî Verify-pass leak check runs *after* egress to the provider ‚Äî **High (fix now)**
**Files:** `app.py:387-389` (verified: `send_messages` at 387, `_verify_or_block` at 389), `app.py:420-424` (chat completions).
**Problem:** The blinded payload is sent upstream **before** `verify_pass` inspects the outbound blob for known real values. So the "block on leak" is *detection after egress*, not prevention. If L2 misses a known real value (casing/normalization), it reaches the provider and the client merely gets a 503 afterward. Invariant #1 ("no real entity ever reaches the provider") is not enforced pre-egress.
**Decision:** run the outbound leak assertion *before* `upstream.send_*`. (The post-restore surrogate-resolution check stays after, since it needs the response.) This splits verify-pass into a pre-egress leak gate and a post-restore resolution gate.

### SEC-6 ‚Äî Streaming path skips verify-pass entirely ‚Äî **High/Medium (fix before relying on streaming)**
**Files:** `app.py:381-385` (`/v1/messages` stream branch returns `StreamingResponse(_stream_restored(...))`, never calls `_verify_or_block`).
**Problem:** The streaming restorer is closed-world with a tail buffer (good), but there's no post-restore assertion for the streamed exchange. L3 fail-closed *does* run before streaming (`app.py:375-379`), so novelty blocking is preserved, but the verify safety net is absent.
**Decision:** thread a streaming verify (e.g. assert over the accumulated restored text as it flushes, or a terminal check) so the streaming path has the same net as the buffered path ‚Äî most naturally combined with `SEC-5`'s pre-egress split.

### SEC-7 ‚Äî Default L3 forwards novel entities unprotected (fail-*open*) ‚Äî **Medium (blocker before relying on novelty protection)**
**Files:** `app.py:141-151` (`_NullAdjudicator`), `app.py:170-181` (`get_l3_detector` defaults `None`, `get_l3_adjudicator` defaults `_NullAdjudicator`).
**Problem:** In the shipped default wiring there is no real L3: the detector is `None` (engine never runs L3) and the adjudicator classifies everything as "not an entity." A novel unknown entity is neither protected nor blocked ‚Äî it **egresses**. Fail-closed *block* only fires when an adjudicator raises `L3Unavailable` (tests inject that). So the promise of invariant #4 ("block novel unscanned content") is inert by default. This is an intentional pre-production stub, but the shipped default silently forwards novel entities.
**Decision:** wire a real Ollama adjudicator (this also unblocks the learning-loop review inbox, which never populates today ‚Äî see `UX-6`), or make the "no L3 available" default itself fail-closed rather than fail-open.

### SEC-8 ‚Äî Re-identify audits only on success; denied/failed attempts leave no trail ‚Äî **Medium (fix now-ish)**
**Files:** `app.py:876-898` (`reidentify_surrogate`), contrast the search endpoint which audits misses (`app.py:1279-1286`).
**Problem:** The audit record is written only after `transit.decrypt` succeeds. The 403 (no role), 404 (surrogate not in workspace), 503 (Transit unconfigured), and decrypt-exception paths are all **unaudited**. An attacker probing for surrogates or attempting unauthorized re-identification leaves no trace ‚Äî inconsistent with ADR-0018's own "audit even misses" principle. Invariant #3 ("every decrypt is always audited") holds for successful decrypts but not for the *attempt* surface.
**Decision:** audit the *attempt* (before the role check) or at minimum audit 403/failed-decrypt as denied events.

### SEC-9 ‚Äî Relationship-edge CRUD has no RBAC ‚Äî **Medium/High (blocker before multi-user)**
**Files:** `app.py:907-935` (`create_relationship_edge`), `app.py:938-948` (`delete_relationship_edge`); the SPA even notes "edge CRUD (no role required, #27)" (`spa.py:1149`).
**Problem:** No `_require_role`. Any caller can add/remove `employer`/`subsidiary_of` edges, which drive the coherent surrogate world and disambiguation. Review-inbox endpoints (`app.py:441-505`) are likewise unauthenticated **and** return plaintext real values (`item.real`, `app.py:451`) ‚Äî that's both an authz gap and a plaintext leak.
**Decision:** gate edge CRUD and the review inbox behind a role once `SEC-1` gives identity meaning; decide whether inbox `real` values need encryption-at-rest given they're pre-review novel entities.

### SEC-10 ‚Äî Transit ETL leaves real values in plaintext columns too ‚Äî **Medium (blocker before relying on at-rest encryption)**
**Files:** `etl.py:166-213` (`run_etl_with_transit`), `migrations.sql:115-125`.
**Problem:** The "encrypted" ETL loads plaintext columns first, then *adds* ciphertext columns but **never nulls** the plaintext `canonical_name`/`value`/`*_variations.value` columns. After it runs, the DB holds real values in **both** plaintext and ciphertext. Invariant #2 is not actually achieved by the Transit path as written. (ADR-0008 defers this, but the code presents itself as the encrypted path.)
**Decision:** null the plaintext columns after ciphertext population, or drop them in a follow-on migration; make the "encrypted" path actually encrypted-at-rest.

### SEC-11 ‚Äî Infra dev posture ‚Äî **Medium (dev-only)**
**Files:** `docker-compose.dev.yml:9` (`VAULT_DEV_LISTEN_ADDRESS: 0.0.0.0:8200`), `config.py:19` (`openbao_addr = http://localhost:8200`), `transit.py` (plain `httpx.Client`).
**Problem:** OpenBao binds all interfaces with the well-known dev root token ‚Üí full decrypt reachable on a shared network. Transit token + decrypted plaintext traverse plain HTTP. No CORS is configured (safe same-origin default ‚Äî but keep it locked, since there's no auth to fall back on). The FastAPI bind isn't pinned in-repo.
**Decision:** bind to `127.0.0.1` for dev; require HTTPS if OpenBao is ever remote; document loopback-only until `SEC-1` lands.

### SEC-12 ‚Äî Audit log is in-memory, unbounded, non-durable, unsigned ‚Äî **Medium (blocker before compliance reliance)**
**Files:** `policy.py:77-84` (`AuditLog` is a plain in-process list).
**Problem:** Records are lost on restart, not persisted, not tamper-evident. No delete endpoint (good), but a crash discards the whole trail. Persistence is explicitly deferred (ADR-0007/0008).
**Decision:** durable, append-only, ideally tamper-evident audit store before the audit log is relied on for GDPR/compliance evidence (a stated driver in CONTEXT.md).

### SEC-13 ‚Äî OpenAI chat-completions ignores `stream:true` ‚Äî **Medium**
**Files:** `app.py:395-427`, `upstream.py:41-48`.
**Problem:** `chat_completions` never checks `payload.get("stream")` and always calls the non-streaming path ‚Üí `response.json()`. A client requesting `stream:true` on the OpenAI endpoint gets a 500 when upstream returns SSE. Not a leak, but an unhandled path ‚Äî confirm it can't fall through to forwarding un-restored SSE.
**Decision:** either support streaming here (mirror the Anthropic path) or reject `stream:true` explicitly with a clear error.

### SEC-14 ‚Äî CLI prints real names to stdout ‚Äî **Low (by design)**
**Files:** `cli.py:187-205`. Expected for a local curation tool, but real-value output can reach shell history / CI logs. Note only.

## 3. Usability & operability findings

### UX-1 ‚Äî A fresh install is a locked, empty room ‚Äî **Strong**
**Files:** `app.py:82` (`_mapping` seeded), `app.py:110` (`_entity_graph = EntityGraph()` ‚Äî **empty**), `app.py:99` (`_rbac = RbacRegistry()` ‚Äî **empty**), `app.py:104` (`_reidentify_store` ‚Äî **empty**).
**Problem:** Three wiring gaps combine so a first-time operator following the README gets a management app that shows nothing and permits nothing:
1. **Entity graph never seeded into the UI store.** `_mapping` is built from the vendored seed, but `_entity_graph` is left empty and nothing loads the seed into it ‚Äî so `/ui/org-graph` and `/ui/entity-list` render an empty workspace even though the proxy is protecting those same names.
2. **RBAC is un-bootstrappable.** Granting a role requires the caller to already hold `admin`; there is no seed admin, CLI, or env-var bootstrap. Merge, rename, reveal, real-name search, audit view, and role-grant all 403 forever out of the box.
3. **Re-identify store starts empty** (populated only by the never-invoked Postgres ETL) ‚Üí Reveal 404s even with a role.
**Decision:** load the vendored seed into `_entity_graph` at startup (same as `_mapping`); grant a bootstrap `admin`/`re-identifier` from an env var (e.g. `BLINDFOLD_BOOTSTRAP_ADMIN`); populate the re-identify store. (Related to `ARCH-3`.)

### UX-2 ‚Äî No runnable entry point; ASGI server isn't even a dependency ‚Äî **Strong**
**Files:** `pyproject.toml` (no `uvicorn`/`hypercorn`, no `[project.scripts]`, no `__main__`), `README.md:107-114`, `cli.py:211-215` (`main()` raises `SystemExit("... requires a curation store; production wiring ... lands in a follow-up slice")`).
**Problem:** The README says to point `ANTHROPIC_BASE_URL` at `localhost:8000` but never gives a run command, and there's no way to launch the server from the documented steps. The CLI deliberately aborts.
**Decision:** add `uvicorn` to deps, a `[project.scripts] blindfold = "blindfold.__main__:main"`, and a real quickstart (compose up OpenBao + Postgres, run ETL, `blindfold serve`, point the client).

### UX-3 ‚Äî `spa.py` is 1,888 lines of three embedded HTML docs with duplication ‚Äî **Worth exploring (Strong for the merge/reveal de-dup)**
**Files:** `spa.py` ‚Äî `_HTML` (`:58`), `_ORG_GRAPH_HTML` (`:154`), `_ENTITY_LIST_HTML` (`:973`).
**Problem:** Three full HTML documents (CSS + `<script type="module">`) as Python triple-quoted strings, with no build step, linter, type-checker, or syntax highlighting over ~1,600 lines. Heavy duplication: merge dialog, reveal-badge, workspace-select bootstrap, and DOM helpers exist in both big SPAs in divergent forms. Endpoint constants are declared **twice** (Python module consts used only by tests at `spa.py:18-27`, and JS `const`s the browser actually runs) ‚Äî so a path change needs 2-3 edits and tests assert the copy the browser *doesn't* use. Escaping hazard: `esc()` exists only in the entity-list SPA; the org-graph SPA has no escape helper. Off-ADR: ADR-0011 promises a "React/Vue SPA"; only the inbox page uses Vue (via CDN), the other two are vanilla JS.
**Decision:** incremental ‚Äî first extract shared CSS/JS fragments (kill the merge/reveal/select/endpoint duplication); medium-term move pages to `.html` files served via `HTMLResponse` to regain tooling; long-term honor ADR-0011 with a built SPA. Pick **one** reveal/merge UX for both pages.

### UX-4 ‚Äî Inconsistent, partly-broken interaction & error states ‚Äî **Worth exploring**
**Files:** org-graph uses custom modals (`spa.py:610-685`); entity-list reimplements the same actions with browser `confirm()`/`alert()` (`spa.py:1678,1686-1687,1464,1548`). Silent failures: `buildDialogRevealBadge` swallows 403/non-OK with no message (`spa.py:1725-1726`). Non-atomic edge re-target: `retargetEdge` DELETEs then POSTs; a failed POST **silently loses the edge** (`spa.py:1515-1571`). Merge winner/loser decided by drag direction / check order, correctable only via a `‚áÑ Swap` button (`spa.py:886-887`) ‚Äî easy to retire the wrong entity.
**Problem:** Same conceptual actions render two different UIs; several paths fail silently or via `alert()`; one path is destructively non-atomic.
**Decision:** unify the reveal/merge UX (with `UX-3`); add a server-side atomic re-target/PATCH endpoint; replace silent catches/`alert()` with consistent inline error text; make merge winner explicit before commit.

### UX-5 ‚Äî Entity-list ceiling blanks the list; probe-on-load floods the audit ‚Äî **Worth exploring**
**Files:** `spa.py:1157` (`ENTITY_LIST_CEILING = 150`), `spa.py:1227-1230` (over the ceiling ‚Üí `allRows = []`), `spa.py:1199-1212` (`detectSearchCapability` probes `entities/search?q=__probe__` every load).
**Problem:** A workspace with >150 entities blanks the entire list (no pagination; surrogate-substring filter is client-side so it can't reach rows past the ceiling). And per ADR-0018 the probe emits an `entity-list-searched` audit record on every page load for role-holders ‚Äî flooding the audit log with `hit_count=0` events, the opposite of ADR-0018's "deliberate unmasks only" goal.
**Decision:** add server-side pagination/filtering to the list endpoint; detect re-identifier capability from a dedicated capability/roles endpoint instead of a `q=__probe__` search.

### UX-6 ‚Äî Learning loop never populates in a real run ‚Äî **Worth exploring**
**Files:** `app.py:141-151,180-181` (`_NullAdjudicator`), `infra/docker-compose.dev.yml` (no Ollama service).
**Problem:** With `_NullAdjudicator` reporting "no novel entities" for everything and no Ollama wired, the review inbox essentially never populates outside tests. The learning loop (a core CONTEXT.md concept) is dark. (Same root as `SEC-7`.)
**Decision:** wire Ollama (compose service + real adjudicator); folds together with `SEC-7`.

### UX-7 ‚Äî SPA behaviour is untested; browser gate is a promise, not a test ‚Äî **Worth exploring**
**Files:** `tests/test_*spa*.py` ‚Äî substring assertions over the served HTML string (e.g. `test_graph_editor_spa.py:102` `assert "merge-dialog" in body or "merge_dialog" in body or "mergeDialog" in body`).
**Problem:** These verify marker strings exist in a 500-line string; they can't catch a broken handler, bad fetch URL, or JS exception, and the `or`-chains pass almost anything. No Playwright tests exist in `tests/` despite the `browser-verify` harness being referenced ‚Äî so merge-swap, ceiling-blanking, gated-reveal, and silent-403 behaviour is entirely unverified. (The JSON-API-seam tests and the privacy/vocab assertions *are* strong ‚Äî keep those.)
**Decision:** add a small committed Playwright suite driving `/ui/entity-list` and `/ui/org-graph` against a seeded app, asserting authorized-only reveal, audit-on-reveal, and no real-name egress (the `browser-verify`/`leak-audit` properties).

### UX-8 ‚Äî Docs drift undermines trust ‚Äî **Strong (cheap, high impact)**
**Files:** `docs/DESIGN.md:3,38,42-56,128`, `README.md:96,120-126,149-150,161`, ADR-0001, ADR-0011.
**Problem:** `DESIGN.md` says "Implementation not started" (badly understates a working proxy + RBAC + merge + 3 SPAs). Claims LiteLLM (ADR-0001 title, README dep) ‚Äî **not in `pyproject.toml`**; the gateway is hand-rolled FastAPI + httpx. Claims Ollama/Presidio/Faker/content-cache ‚Äî none present (L3 is a stub, L1 is hand-rolled regex, surrogates hand-rolled). "Warm start seeds the graph" ‚Äî seeds `_mapping` only, not the UI's `EntityGraph` (`UX-1`). ADR-0011 "React/Vue SPA" ‚Äî only the inbox uses Vue.
**Decision:** reconcile docs to the hand-rolled reality (or add ADRs recording the pivots away from LiteLLM/Presidio); fix the "warm start" and "implementation not started" claims. This is the cheapest trust win in the doc.

### UX-9 ‚Äî Dev-harness JS pollutes the repo root ‚Äî **Worth exploring**
**Files:** root `package.json`, `package-lock.json`, committed `node_modules/`.
**Problem:** A Python-only project (`pyproject.toml`, `uv.lock`) ships a committed root `node_modules/` that only serves the `.sandcastle/` agent harness. Confusing ‚Äî implies a JS build that doesn't exist.
**Decision:** move the sandcastle `package.json` under `.sandcastle/`, gitignore `node_modules/`, and note in the README that root JS tooling is dev-harness-only.

## 4. Sandcastle (agentic-dev harness) findings

The `.sandcastle/` loop (`main.mts`, prompt files, Dockerfile) is a plan‚Üíimplement‚Üíreview‚Üí(web-verify)‚Üímerge orchestration with a genuinely well-designed **fail-closed merge gate**: a branch reaches main only on positive attestations from the implementer, an adversarial Opus reviewer, and (for SPA branches) a Playwright browser gate; issue-lifecycle side-channels correctly fail-*open* so a `gh` hiccup never influences the gate. Findings below are about the gaps in that model.

### SC-1 ‚Äî The merge phase is itself ungated ‚Äî **Strong**
**Files:** `.sandcastle/main.mts:700-713` (merge agent), `merge-prompt.md`.
**Problem:** Every branch is independently reviewed, but the **merge agent** then resolves conflicts and is instructed to "fix the issues if tests fail" (`merge-prompt.md:11-13`) ‚Äî mutating code that **no reviewer re-audits**. The merged-label / issue-close / branch-reap lifecycle (`main.mts:720-734`) runs unconditionally after the merge call, without checking the merger's own completion signal. So a conflict resolution or test-fix by the merger can reach `main` without passing the leak-audit gate that every other change must clear. For a fail-closed privacy product this is the one hole in an otherwise fail-closed loop.
**Decision:** re-run the reviewer (or at least the leak-audit) on the post-merge result before labeling merged; gate the lifecycle on the merger's completion signal.

### SC-2 ‚Äî Shell-injection surface via planner-generated branch names ‚Äî **Worth exploring (Medium)**
**Files:** `.sandcastle/main.mts:104-107` (`branchTouchesSpa` builds an `execSync` **shell string** interpolating `branch`), vs. everything else which correctly uses `execFileSync` (no shell).
**Problem:** Branch names come from the planner's JSON output, which is derived from **untrusted issue text** (bodies + all comments, see `SC-3`). Although the plan schema constrains shape, the branch string is interpolated into a shell command. A crafted branch value could inject shell. Every other git/gh call in the file already uses `execFileSync` ‚Äî this is the one that doesn't.
**Decision:** switch `branchTouchesSpa` to `execFileSync("git", ["diff", "--name-only", `${TARGET_BRANCH}...${branch}`, "--", ...SPA_PATHS])`. Cheap, removes the surface entirely.

### SC-3 ‚Äî Untrusted issue text flows verbatim into agent prompts ‚Äî **Worth exploring (Medium)**
**Files:** `.sandcastle/plan-prompt.md:7` (`gh issue list ... --json ...body,comments`), `implement-prompt.md` (pulls issue + comments).
**Problem:** Issue **bodies and all comments** are piped into the planner and implementer prompts. In a public or multi-contributor repo, a hostile issue comment is a prompt-injection vector against an autonomous agent that writes code and gates merges. The planner prompt has good HITL guardrails (never auto-work a privacy-contract change), but those are instructions to a model, not a hard boundary.
**Decision:** decide the trust boundary for issue text (restrict autonomous pickup to the `Sandcastle` label applied by a trusted maintainer ‚Äî already the policy per memory; make it explicit that label application is the trust gate), and consider stripping/sandboxing comment text.

### SC-4 ‚Äî Model-comment drift in the orchestrator ‚Äî **Low (fix now, cheap)**
**Files:** `.sandcastle/main.mts:5` (header comment says "An **opus** agent analyzes open issues") vs. `main.mts:62,383` (`MODEL_PLAN = "claude-sonnet-4-6"`), and `main.mts:368` ("opus, for deeper reasoning") vs. the Sonnet config.
**Problem:** The header/inline comments claim the planner runs on Opus; the config runs it on Sonnet. Misleading for anyone tuning per-role cost (and the memory note "Opus for the privacy gates" is honored only for the reviewer, which is correct ‚Äî but the planner comments contradict the code).
**Decision:** fix the comments to match (planner = Sonnet; only the reviewer gate is Opus), or bump the planner if deeper dependency-graph reasoning is actually wanted.

## 5. Ubiquitous-language gaps (candidate `CONTEXT.md` terms)

Concepts the code and the findings above lean on that `CONTEXT.md` does not yet name. Per `CONTEXT.md`'s own rule, add these via `/grill-with-docs` (don't invent synonyms). Listed strongest first.

- **Identity (Principal)** ‚Äî the authenticated caller a role attaches to; the subject of a role assignment. Biggest gap: the whole RBAC model and `SEC-1` revolve around "who is making this management request," but the glossary names `Workspace` and roles without ever naming the subject. Today it's a bare `x-blindfold-identity` header. Grill this *with* `SEC-1` ‚Äî the term and the auth fix are the same conversation.
- **Role (controlled vocabulary: `admin`, `re-identifier`)** ‚Äî roles carry the same drift risk already guarded for `Relationship`. `re-identifier` is mentioned inside `Re-identify`/`Workspace`; `admin` lives only in code (`grant_workspace_role`); there is no consolidated entry stating the closed role set. Give it the "controlled vocabulary, not free text" treatment `relation` gets.
- **Retired surrogate** ‚Äî currently defined inline inside `Merge` ("retired ‚Äî kept restorable, never deleted") but it is a distinct persistent state with its own `retired_surrogates` table. Promote to a one-line entry so "retired ‚âÝ deleted, still restorable" is findable.
- **Novel entity** ‚Äî an entity not yet in the entity graph; the trigger for the provisional/learning-loop path. `Provisional surrogate` and `Review inbox` both define themselves in terms of it, but the base term is never defined. Relates to `SEC-7`/`UX-6`.
- **Egress** ‚Äî the moment a payload leaves the machine. Used throughout the leak-audit property ("no real entity egresses," "pre-egress"), and `SEC-5` hinges on the timing invariant that verify-pass must run *before* egress. `CONTEXT.md` says "before it leaves the machine" but never names the moment.

Considered and rejected as implementation/test-harness concerns, not ubiquitous language: bootstrap admin, capability probe, egress oracle / stub upstream.
